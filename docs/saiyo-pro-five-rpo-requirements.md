# 採用PRO / FIVE RPO LINE 要件

## 目的

FIVE 公式LINEを最初の企業向けテンプレートとして完成させ、以後の企業公式LINEへ横展開できる形にする。

FIVE は単独の特注実装ではなく、`企業公式LINE + 採用PRO公式LINE` の標準パターン第一号として扱う。

## 全体フロー

1. 候補者が Airワーク等から FIVE の求人へ応募する。
2. FIVE 公式LINEへ誘導する。
3. FIVE 側で選考状況、問い合わせ、選考結果を扱う。
4. 見送りになった候補者へ、採用PRO公式LINEへの登録導線を送る。
5. 採用PRO公式LINEで希望条件などの最低限情報を段階的に取得する。
6. 採用PRO側で求人準備ができ次第、条件に合う求人を通知する。
7. 候補者が求人へ応募し、採用PRO for Biz 側の企業とマッチングする。

## アカウント分離

LINE userId は公式LINEごとに別コンテキストとして扱う。

- FIVE 公式LINE: RPO選考、選考結果通知、見送り後の採用PRO案内
- 採用PRO公式LINE: 候補者情報収集、求人案内、応募導線
- 採用PRO for Biz: 企業側求人・応募者管理

同じ人物でも、FIVE 公式LINEと採用PRO公式LINEの `friends` レコードは分離する。実装上は `line_account_id` を必ず見る。

## FIVE 公式LINEで必要な機能

### 初期構築

- LINE account: `five-rpo`
- 表示名: `FIVE公式LINE`
- 流入ref: `five-rpo-rejection`
- タグ: `FIVE不採用後 採用プロ案内`
- traffic pool: `main`

### 自動応答

- `選考状況`: 選考状況確認中の案内
- `問い合わせ`: 担当者確認中の案内
- `他の求人`: 採用PRO公式LINEへの登録案内

### 選考結果通知

管理/運用側から以下を送れるようにする。

- `reject`: 見送り通知 + 採用PRO公式LINE登録案内
- `pass`: 書類選考通過通知
- `hold`: 選考状況確認中の案内

送信履歴は `messages_log` に残し、KPI集計できるよう `source` を結果別に保存する。

## 採用PRO公式LINEで取得する情報

候補者のストレスを下げるため、最初から長いフォームを出さない。必要最低限を段階的に聞く。

初期入力で優先する項目:

- 現在のポジション
- 希望職種
- 希望勤務地
- 希望雇用形態
- 転職意欲または時期
- 連絡可能な時間帯

後続で取得する項目:

- 職務経歴書
- 詳細な職歴
- スキル
- 希望条件の詳細

## 職務経歴書

職務経歴書を持っている人には、PDF またはリンクで送ってもらう。

職務経歴書を持っていない人には、LINE上の入力情報から職務経歴書PDFを作れる導線を用意する。これは候補者の応募ハードルを下げるための重要機能として扱う。

初期メッセージでは次のように案内する。

> 職務経歴書をお持ちの場合は、PDFまたはリンクでお送りください。
> お持ちでない場合は、質問に答えるだけで職務経歴書を作成できます。

ただし、FIVE 公式LINE初期構築では「見送り後の採用PRO誘導」までを先に通す。職務経歴書生成は、採用PRO公式LINE側の段階入力フローに組み込む。

## 初期KPI

FIVE から採用PROへ送客する検証では、まず以下を追う。

- 見送り通知数
- 採用PRO公式LINE追加数
- 条件回答完了数
- 求人案内数
- 求人応募数

最初の優先KPIは `見送り通知数 -> 採用PRO公式LINE追加数 -> 条件回答完了数`。

## 横展開時の可変項目

企業ごとに変えるもの:

- `line_account_id`
- LINE Channel ID / access token / secret
- 表示名
- 流入ref
- タグ名
- 見送り通知文面
- 通過通知文面
- 問い合わせ文面

企業ごとに変えないもの:

- `line_account_id` によるアカウント分離
- 見送り後に採用PRO公式LINEへ送る構造
- 採用PRO側で段階的に最低限情報を取得する構造
- KPI の基本定義
